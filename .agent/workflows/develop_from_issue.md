---
description: Issue URLから開発を開始し、プルリク作成までを行うワークフロー
---

1. **Issue URLの取得と確認**
   - ユーザーに処理対象のGitHub Issue URLを尋ねる（未提供の場合）。
   - `gh issue view <url>` を実行して内容を確認する。
   - 要件に不明点がある場合は、ユーザーに**必ず質問する**。

2. **ブランチの作成**
   - ブランチ名は `<topic>/#<issue-number>_<branch-name>` の形式で提案し、ユーザーの承認を得る。
     - 例: `fix/#1201_resolve-css-bug`, `feat/#123_add-login-page`
   - `git checkout -b <branch-name>` でブランチを作成する。

3. **実装プロセス（反復）**
   - **SSOTの維持**: Issueを単一の真実の情報源（SSOT）として扱う。
   - **計画・要件の変更**: `gh issue comment <issue-number> --body "..."` で記録する。
     - 計画の調整、技術的制約の判明、要件変更時など。
   - **進捗と判断の記録**: 重要な判断やマイルストーン到達時にコメントを追加する。
   - **実装**: コードを変更する。
   - **コミットとプッシュ**:
     - コミットメッセージにはIssue番号を含める（例: `fix: resolve css bug #123`）。
     - 定期的にリモートにプッシュする。

4. **プルリクエストの作成**
   - 実装と検証が完了したら、`gh pr create` を使用してプルリクエストを作成する。
   - 本文に `Fixes #<issue-number>` を含める。

## 重要なガイドライン

- **質問する**: 要件が曖昧な場合は勝手に進めず、必ず質問して明確にする。
- **SSOT**: Issueの内容を常に最新の状態に保つよう心がける。
- **GitHub CLI**: GitHub操作には `gh` コマンドを使用する。
- **承認**: ユーザーの明確な承認があるまで実装（コード変更）に移らない。
- **参照**: コミットやPRには必ずIssue番号を紐付ける。
